# 14-后端落地规划

> 目标：把当前已经搭好的 NestJS + Prisma 后端骨架，推进成真正可用的业务后端，而不是停留在 scaffold / mock 阶段。

---

## 1. 当前状态

当前后端已经具备这些基础：

- NestJS 工程骨架
- 全局鉴权、异常处理、返回包装
- Prisma 数据模型草案
- 身份体系骨架
- 商品、购物车、订单、支付、售后、后台相关接口壳子

但目前仍有一个关键问题：

- 大量业务接口还在返回 scaffold / mock 数据
- 真实数据库读写还没有接通
- 核心闭环没有真正跑起来

所以当前状态可以定义为：

**后端已进入可开发阶段，但还不是可交付阶段。**

---

## 2. 总原则

后端按下面顺序推进：

1. 先让身份和权限可用
2. 再让基础数据可查可写
3. 再让交易闭环跑通
4. 最后补后台治理和扩展能力

原则上不在第一轮同时做：

- 秒杀
- 拼团
- 分销
- 积分全量规则
- 复杂财务结算

这些都放在核心交易闭环之后。

---

## 3. 阶段规划

### 阶段 0：后端可启动

目标：后端工程可稳定启动，基础设施齐全。

要做的事：

- Prisma 连接真实数据库
- 配置 migration 流程
- 补齐 `.env` / `.env.local` 约定
- 确认 Redis、对象存储、微信支付配置入口
- 确认统一返回格式和错误码
- 确认日志输出和异常追踪

验收标准：

- 后端可以启动
- 数据库连接成功
- 基础接口能返回统一格式

---

### 阶段 1：身份与权限

目标：让用户、商家、管理员的身份体系真实可用。

要做的事：

- 微信登录
- 游客 `guest` 身份签发
- 用户 token 签发与刷新策略
- 管理员登录
- 角色权限控制
- `@Public()` / `@Roles()` 的统一落地
- 登录态查询接口
- 登录后访客数据合并规则

验收标准：

- C 端可以登录
- B 端可以识别商家身份
- 后台可以登录并限制权限
- 未登录用户和登录用户行为可区分

---

### 阶段 2：基础数据

目标：把最常用的查询和维护能力接到数据库。

要做的事：

- 分类查询
- 商品列表、商品详情
- SKU 查询
- 商品上下架
- 用户地址管理
- 商家入驻
- 商家资质上传
- 文件上传接真实存储

验收标准：

- 首页、分类页、详情页能从数据库取数
- 商家入驻和商品基础资料可落库
- 上传文件不再是 mock URL

---

### 阶段 3：购物车与下单

目标：让交易链路开始形成闭环。

要做的事：

- 加入购物车
- 修改数量
- 勾选与删除
- 结算页预览
- 创建订单
- 订单商品快照
- 运费与优惠计算

验收标准：

- 用户能从商品页进入购物车和结算
- 可以生成真实订单记录
- 订单金额计算不依赖前端手写

---

### 阶段 4：支付与订单状态

目标：把支付和订单状态流转跑通。

要做的事：

- 微信支付下单
- 支付回调验签
- 支付记录入库
- 订单支付成功状态更新
- 订单取消、超时关闭
- 订单完成确认

验收标准：

- 支付链路有真实状态变化
- 回调是幂等的
- 订单状态和支付状态能对应起来

---

### 阶段 5：发货与售后

目标：把履约和售后闭环补完整。

要做的事：

- 商家发货
- 物流信息记录
- 售后申请
- 商家处理售后
- 平台仲裁
- 退款状态流转
- 库存/资金回滚规则

验收标准：

- 订单能发货
- 售后能流转
- 退款有明确状态和结果

---

### 阶段 6：后台治理

目标：让平台运营和审核动作可用。

要做的事：

- 管理员管理
- 商户审核
- 商品审核
- Banner 配置
- 活动配置
- 运费模板
- 数据看板
- 操作日志
- 活动商品规则配置入口

验收标准：

- 后台可审核平台核心对象
- 运营配置可维护
- 审计记录可追踪
- 活动主线至少能完成“活动 + 商品 + 规则”三段式配置

---

### 阶段 7：扩展能力

目标：在核心闭环稳定后，再加增长模块。

要做的事：

- 优惠券
- 积分
- 秒杀
- 拼团
- 分销
- 团长体系
- 自提点
- 营销中心收敛与集中化整理，避免把业务拆散到很多小文件里
- 先把活动商品规则做成最小闭环，再逐步扩展到更完整的活动引擎

验收标准：

- 扩展模块不破坏核心交易链路
- 新规则只是在已有订单体系上叠加
- 核心营销业务保持集中，后续扩展时仍能快速定位创建、发布、下线和校验逻辑
- 活动商品规则已经可独立配置、可独立查询、可独立回滚

---

## 4. 每阶段的固定开发顺序

每个模块都按同样顺序做：

1. 数据表
2. DTO
3. Service
4. Controller
5. Guard / 权限
6. 单元测试
7. 联调验证
8. 文档更新

补充约束：

- 业务主线优先集中在一个模块内，不要一开始就按“看起来很清晰”的方式拆出过多文件
- 像营销活动这类流程，优先把创建、编辑、发布、下线、校验写在同一个领域模块里
- 只有库存、价格计算、状态机这类明显独立且复用度高的能力，再抽成少量子服务
- 这样能减少文件跳转和业务上下文分散，后续排查问题也更直接

不要反过来先写页面再补接口，否则会反复返工。

## 6. 里程碑

### 里程碑 A：后端可用

结果：

- 后端可启动
- 数据库可连通
- 身份可用

### 里程碑 B：核心交易闭环

结果：

- 能浏览商品
- 能加购
- 能下单
- 能支付
- 能发货

### 里程碑 C：运营可用

结果：

- 后台可审核
- 配置可维护
- 售后可处理

### 里程碑 D：扩展可接入

结果：

- 营销和分销能力可以逐步上线

---

## 7. 风险点

- Prisma schema 和真实业务状态可能还会有字段差异
- 微信登录和支付回调需要严格验签与幂等
- 订单状态、支付状态、退款状态必须统一定义
- 访客态数据合并规则如果晚定义，后面会返工
- 一旦把 mock 页面接到真实后端，字段不一致会暴露很多问题

---

## 8. 验收标准

后端真正可用时，至少满足：

- 用户能登录
- 访客能拿到临时身份
- 商品和分类来自数据库
- 购物车和订单可落库
- 支付和回调能闭环
- 商家能发货
- 售后能流转
- 后台能审核和配置

---

## 9. 结论

当前不是“继续堆功能”的阶段，而是“把核心闭环接真”的阶段。

最稳路线是：

**身份 -> 商品 -> 购物车 -> 订单 -> 支付 -> 发货 -> 售后 -> 后台治理 -> 扩展功能**
