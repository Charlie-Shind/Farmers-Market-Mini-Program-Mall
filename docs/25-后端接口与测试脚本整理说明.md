## 3. 测试脚本现在怎么分

### 3.1 接口可用性烟测

新增脚本：

```bash
cd backend
npm run smoke:api
```

默认只跑只读公开接口：

- `/api/identity/auth/status`
- `/api/identity/auth/anonymous`
- `/api/app/categories`
- `/api/app/products?pageSize=3`
- `/api/app/quick/flash-sale/active`
- `/api/app/quick/group-buy/products?pageSize=3`

如果要把带 token 的只读接口也跑上：

```bash
npm run smoke:api -- --mode=auth
```

会额外检查：

- `/api/identity/auth/me`
- `/api/app/favorites`
- `/api/app/cart`
- `/api/app/orders`
- `POST /api/app/quick/group-buy/nearby`

脚本位置：

- `backend/scripts/smoke-api.ts`

---

### 3.2 单条请求看原始返回

如果你只想盯住某个接口的原始响应，优先用单条请求脚本。它会把状态码、耗时和响应体完整打印出来。

```bash
cd backend
npm run api:request -- --method=GET --path=/app/products?pageSize=3
```

带 token 的例子：

```bash
cd backend
npm run api:request -- --method=POST --path=/app/orders/preview --token=YOUR_TOKEN --body='{"cartIds":[1],"addressId":1,"deliveryType":1}'
```

常用参数：

- `--baseUrl=http://127.0.0.1:6002/api`
- `--method=GET|POST|PUT|PATCH|DELETE`
- `--path=/api/...`
- `--token=...`
- `--query='{"pageSize":3}'`
- `--body='{"a":1}'`
- `--headers='{"x-demo":"1"}'`

脚本位置：

- `backend/scripts/api-request.ts`

---

### 3.3 流程测试看完整链路

如果你想按业务流程一步一步看接口返回，优先用 `backend/test/api-flow.ts`。

```bash
cd backend
npm run test:api:flow -- --section=full
```

只看某一段：

```bash
cd backend
npm run test:api:flow -- --section=order
```

如果要把每一步响应内容也打印出来，加 `--dump`：

```bash
cd backend
npm run test:api:flow -- --section=order --dump
```

脚本会覆盖：

- 公开读接口
- 登录与商家入驻
- 商品与草稿
- 活动、订单、退款
- 商家与后台管理

### 3.4 一条跑通主流程

如果你要一条“可以直接跑通”的主流程，优先用这个脚本：

```bash
cd backend
npm run test:api:happy
```

它会自己完成：

- 买家登录
- 商家申请与审核
- 商品创建、审核和上架
- 买家浏览、加车、下单和支付
- 商家接单、发货
- 买家确认、评价和退款
- 商家处理退款

想看每一步返回值时，加 `--dump`：

```bash
cd backend
npm run test:api:happy -- --dump
```

现有脚本继续保留，作为联调和回归的测试数据入口：

- `scenario:test:*`
- `scenario:payment:*`
- `scenario:flash:*`
- `scenario:group:inject`

## 4. 清理文件的边界

这些目录或文件属于可清理生成物，不应作为源码维护：

- `backend/dist/`
- `backend/.idea/`
- `backend/tsconfig.tsbuildinfo`
- 根目录或子项目里的日志、pid、临时文件

这些内容已经在 `.gitignore` 中被排除，日常不要把它们当成需要提交的项目内容。

推荐清理方式：

```bash
cd backend
npm run clean
```

如果是 IDE 目录或历史构建目录，直接删除工作区副本即可，不需要改业务代码。

## 5. 后续维护建议

- 新增接口时，先改控制器和对应 domain service，再补 `backend/README.md`
- 新增可复用测试入口时，优先放到 `backend/scripts/` 并注册到 `package.json`
- `docs/02-接口-总览.md` 只做目录，不再写成长篇草稿
- `docs/上手指南/04-API自测清单.md` 只保留“最短可执行验证路径”
