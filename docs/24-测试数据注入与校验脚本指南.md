# 24-测试数据注入与校验脚本指南

> 目标：补齐项目里"测试场景数据"的注入与校验工具链，提供可重复触发、可参数化、带断言的独立脚本，统一收录在 `backend/scripts/` 下。

---

## 1. 为什么需要独立脚本（而不是改 seed.ts）

| 维度 | `prisma/seed.ts` | `scripts/inject-*.ts` |
| --- | --- | --- |
| 触发频率 | 项目初始化一次 | 开发 / 联调 / 测试期间任意次 |
| 幂等性 | 多数段使用 `count() === 0` 守卫 | 支持 `--reset` 强制重置 |
| 参数化 | 改源码 | CLI 参数：`--days=3 --items-per-window=6 --dry-run` |
| 写后校验 | 无 | 内置断言 + 独立 verify 脚本 |
| 影响范围 | 全量 | 仅限本场景数据 |

**结论**：`seed.ts` 负责"项目开局数据"（角色/账号/分类/初始商品）；测试场景数据走独立脚本。

## 3. 限时秒杀注入脚本 `inject-flash-sale.ts`

### 3.1 用法

```bash
npm run scenario:flash:inject                          # 默认：3 天 × 8 场 × 6 件 = 144 件
npm run scenario:flash:inject -- --days=5              # 改天数（1–7）
npm run scenario:flash:inject -- --items-per-window=10  # 改每场件数（1–20）
npm run scenario:flash:inject -- --reset               # 清空旧 window/item/claim 后再注入
npm run scenario:flash:inject -- --dry-run             # 只打印计划，不写库
```

> 前置条件：已执行过一次 `npm run prisma:seed`，确保 `Product` / `ProductSku` 有可售库存。

### 3.2 数据契约

依据 `backend/prisma/schema.prisma` 的 3 张表生成：

#### FlashSaleWindow
| 字段 | 类型 | 注入值 | 说明 |
| --- | --- | --- | --- |
| `label` | string(64) | `HH:MM 限时秒杀` | 例 `10:00 限时秒杀` |
| `startAt` | timestamp | 当日 / 次日 / 后日 整点 | 10/12/14/16/18/20/22/次日 00 共 8 个 |
| `endAt` | timestamp | `startAt + 2h` | 每场 2 小时 |
| `status` | string(16) | 动态计算 | 相对当前时刻：`ENDED` / `ONGOING` / `UPCOMING` |
| `sortOrder` | int | 自增 | 注入顺序 |

#### FlashSaleItem
| 字段 | 类型 | 注入值 | 说明 |
| --- | --- | --- | --- |
| `flashPrice` | decimal(18,2) | `originPrice * (5–8 折随机)` | 永远 < `originPrice` |
| `originPrice` | decimal(18,2) | 取自 `ProductSku.price` | 与 SKU 一致 |
| `totalStock` | int | 60% 充足(50–100) / 25% 紧俏(5–19) / 15% 售罄 | 售罄仅在非 UPCOMING 场次出现 |
| `stockLeft` | int | `totalStock` 或 `0` | 售罄品 = 0 |
| `perUserLimit` | int | 紧俏品 1 / 其他 2 | 紧俏品限购更严 |

#### FlashSaleClaim
注入脚本**不**创建 claim，claim 由 `claimFlashSale` 接口在用户点击"立即抢购"时动态写入。

### 3.3 注入后内置断言（6 条）

| 断言 | 校验内容 | SQL / 查询 |
| --- | --- | --- |
| 窗口数符合预期 | 至少注入预期场次数 | `flashSaleWindow.count()` |
| 商品数符合预期 | 至少注入预期商品数 | `flashSaleItem.count()` |
| 价格合法 | 所有 `flashPrice < originPrice` | `WHERE flash_price >= origin_price` 计数 = 0 |
| 库存合法 | `0 ≤ stockLeft ≤ totalStock ≥ 0` | `WHERE stock_left<0 OR stock_left>total_stock OR total_stock<0` = 0 |
| 场次无重叠 | 任意两个 window 的 `[startAt, endAt)` 不相交 | 自连接 + 区间相交判断 |
| 折扣在 5–8 折 | `flash_price/origin_price ∈ [0.5, 0.8]` | `WHERE ratio NOT BETWEEN 0.5 AND 0.8` = 0 |

任何一条失败 `process.exitCode = 1`。

## 4. 限时秒杀校验脚本 `verify-flash-sale.ts`

### 4.1 用法

```bash
npm run scenario:flash:verify
```

不写库，只查 + 校验，任意时刻可跑（CI / 联调前 / 部署后）。

### 4.2 8 条断言

| # | 断言 | 校验内容 |
| --- | --- | --- |
| 1 | 场次存在 | `flashSaleWindow.count() > 0` |
| 2 | 商品存在 | `flashSaleItem.count() > 0` |
| 3 | 价格合法 | 所有 `flashPrice < originPrice` |
| 4 | 库存合法 | `0 ≤ stockLeft ≤ totalStock` |
| 5 | 场次无重叠 | 自连接判断区间 |
| 6 | 折扣在 5–8 折 | `ratio ∈ [0.5, 0.8]` |
| 7 | 无空场次 | 每个 window 都有 ≥1 个 item |
| 8 | 关联完整 | item 关联的 window/sku/product 均存在 |

附加输出：
- 场次状态分布（UPCOMING / ONGOING / ENDED 各多少）
- 最早 5 场列表（含起止时间、商品数）

## 5. 接口契约（与注入数据对齐）

小程序 `wxapp/miniprogram/services/quick.ts` 调用的 4 个接口，注入数据必须满足其约束才能在前端正常显示。

| 路由 | 入参 | 出参关键字段 | 注入数据需满足 |
| --- | --- | --- | --- |
| `GET /app/quick/flash-sale/windows` | — | `windows[].status: ONGOING\|UPCOMING\|ENDED` | 任意时段三类状态都要有 |
| `GET /app/quick/flash-sale/items` | `windowId, page, pageSize` | `items[].stockLeft, totalStock, flashPrice, originPrice` | `stockLeft ≥ 0`、`flashPrice < originPrice` |
| `GET /app/quick/flash-sale/active` | — | `windows, items` 综合 | 同上 |
| `POST /app/quick/flash-sale/claim` | `{itemId, quantity}` | `{itemId, skuId, flashPrice, orderNo?}` | `stockLeft ≥ quantity` 且 `perUserLimit` 未超 |

相关后端实现位置：`backend/src/common/services/platform-data.service.ts`
- `getQuickFlashSaleActive` (line 9709)
- `getQuickFlashSaleWindows` (line 9825)
- `getQuickFlashSaleItems` (line 9871)
- `claimFlashSale` (line 9960)

## 7. 常见问题

**Q1: `inject` 跑过一次再跑会重复插入吗？**
不会报错，但会叠加数据。要重置请加 `--reset`。

**Q2: 为什么 discount 范围是 5–8 折？**
这是注入脚本的策略区间，可在 `pickStock` / `rate` 计算处调整。如果改动了区间，请同步更新 `inject` 内的断言与 `verify` 的折扣断言。

**Q3: SKU 池只有 63 个，跑多天会有重复商品吗？**
会。脚本用 `(windowSeq * itemsPerWindow + i) % pool.length` 取模轮询，保证分布均匀但允许跨场次复用同一商品。这是当前设计，未来若要"每场不重复"需扩展 SKU 池或换策略。

**Q4: 没连数据库能跑吗？**
不能。脚本依赖 `PrismaClient` + 真实 SQL，需要 `backend/.env.local` 含 `DATABASE_URL`。dry-run 模式仍要连接（用于读取 SKU 池）。
