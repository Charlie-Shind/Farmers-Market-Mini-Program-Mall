# 06-数据-数据库设计

> 核心表、字段、索引、状态枚举与关系说明。

## 1. 文档目标

本文档根据现有需求文档与技术方案，整理出农仓电商平台的核心数据库设计，用于：

- 指导后端接口和 Prisma Schema 设计
- 指导数据库建表与索引设计
- 统一业务状态、字段命名和关联关系
- 为后续扩展营销、分销、自提点等能力预留结构

数据库建议采用 `PostgreSQL`，编码统一为 `UTF8`，时间字段统一使用 `timestamptz` 或 `timestamp` 并明确时区处理方式。

---

## 2. 当前 Prisma 落地状态

当前仓库里的 `backend/prisma/schema.prisma` 已经落地了第一阶段核心业务模型，重点覆盖：

- 用户与角色
- 商家与资质
- 商品、SKU、图片、视频、溯源
- 购物车
- 订单、支付、发货、售后
- 后台账号、后台角色、后台操作日志
- 商家钱包和提现申请

当前还在规划中的扩展模块包括：

- 优惠券
- 活动
- 积分
- 团长分销
- 自提点
- 分佣和提现

因此本文档分成两层看更准确：

- **已落地部分**：可以直接指导 Prisma 和数据库开发
- **扩展规划部分**：保留字段和模型方向，但可在第二阶段补齐

## 3. 设计原则

1. **业务拆分清晰**  
   按用户、商家、商品、订单、支付、售后、营销、分销分域建模。

2. **状态驱动**  
   核心流程使用状态枚举，不用布尔值表达复杂业务。

3. **可扩展优先**  
   预留软删除、审计字段、外部资源地址、JSON 扩展字段。

4. **查询友好**  
   对高频查询字段建立联合索引，例如 `user_id + status`、`merchant_id + status`。

5. **幂等与可追溯**  
   支付、退款、发货、售后等表必须保留业务单号和第三方流水号。

---

## 4. 通用字段规范

建议所有业务表统一包含以下字段：

- `id`：主键，建议 `bigint` 或 `uuid`
- `created_at`：创建时间
- `updated_at`：更新时间
- `deleted_at`：软删除时间，或使用 `is_deleted`
- `status`：业务状态字段
- `remark`：备注字段，按需添加

建议所有外部文件、图片、视频统一存储为 URL 或对象存储 key，不直接存二进制。

- `merchant` 1:N `product`
- `product` 1:N `product_sku`
- `product` 1:N `product_image`

- `product` 1:N `product_trace`
- `user` 1:N `cart`

- `order` 1:N `order_item`

- `order` 1:N `delivery_record`
- `order` 1:N `refund_apply`
- `user` 1:N `user_coupon`

### 6.2 快照字段说明

订单、售后、支付等核心业务需要保存快照字段，避免后续商品名称、价格、地址变更影响历史单据展示。

## 9. 建表补充建议

1. `bigint` 主键优先，便于高并发场景扩展。
2. 金额字段统一使用 `numeric(18,2)`，避免浮点误差。
3. 规格字段建议使用 `jsonb`，适合后续扩展 SKU 属性。
4. 图片、视频、文件统一保存 URL，不直接入库大对象。
5. 所有支付、退款、提现单据必须有唯一业务单号。
6. 订单、售后、支付必须保留快照，禁止依赖实时商品信息回填历史单据。

## 10. 实施顺序建议

建议第一期优先落表：

- 用户体系：`user`、`role`、`user_role`、`user_address`
- 商家体系：`merchant`、`merchant_qualification`、`merchant_wallet`
- 商品体系：`category`、`product`、`product_sku`、`product_image`
- 交易体系：`cart`、`order`、`order_item`、`payment_record`、`delivery_record`
- 售后体系：`refund_apply`
- 营销体系：`coupon`、`user_coupon`

第二期再补：

- `activity`、`activity_product`
- `points_log`
- 分销相关表
- 后台权限体系
