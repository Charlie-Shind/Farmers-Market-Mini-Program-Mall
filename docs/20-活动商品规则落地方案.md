# 20-活动商品规则落地方案

> 目标：先把“活动 + 商品 + 规则”这条最小闭环做通，再逐步扩展到秒杀、拼团、满减、折扣等完整营销能力。

---

## 1. 本阶段目标

先做以下三件事：

1. 活动主表能创建、编辑、列表、下线
2. 活动可以绑定商品和 SKU
3. 活动可以配置规则，规则先覆盖最小必需字段

本阶段不追求完整营销引擎，不先做复杂规则叠加、优先级编排和多活动叠加结算。

---

## 2. 统一业务模型

建议把活动系统收敛成以下三层：

```text
Activity（活动主表）
├── ActivityProduct（活动商品）
└── ActivityRule（活动规则）
```

### 2.1 `activity`

活动主表，负责承接活动基础信息和生命周期。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| merchant_id | bigint | 商家 ID |
| activity_name | varchar(128) | 活动名称 |
| activity_type | smallint | 活动类型 |
| status | smallint | 状态 |
| start_at | timestamp | 开始时间 |
| end_at | timestamp | 结束时间 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

建议状态：

- `1` 草稿
- `2` 待开始
- `3` 进行中
- `4` 已结束
- `5` 已下线

### 2.2 `activity_product`

活动商品表，承接商品和 SKU 绑定。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| activity_id | bigint | 活动 ID |
| product_id | bigint | 商品 ID |
| sku_id | bigint | SKU ID |
| activity_price | numeric(18,2) | 活动价 |
| stock_limit | int | 活动库存/限购库存 |
| limit_per_user | int | 每人限购 |
| created_at | timestamp | 创建时间 |

建议索引：

- `idx_activity_product_activity_id(activity_id)`
- `idx_activity_product_product_sku(product_id, sku_id)`

### 2.3 `activity_rule`

活动规则表，负责承接秒杀、拼团、满减、折扣等差异化配置。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| activity_id | bigint | 活动 ID |
| rule_type | smallint | 规则类型 |
| rule_json | jsonb | 规则详情 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

建议 `rule_type`：

- `1` 秒杀
- `2` 拼团
- `3` 满减
- `4` 折扣

`rule_json` 建议保存的最小字段：

- 秒杀：`flashPrice`、`stock`、`limitPerUser`
- 拼团：`groupPrice`、`requiredCount`、`expireHours`
- 满减：`thresholdAmount`、`discountAmount`
- 折扣：`discountRate`

## 3. 创建流程

商家后台创建活动的推荐顺序：

```text
选择活动类型
→ 填活动基础信息
→ 选择商品 / SKU
→ 配活动规则
→ 设开始和结束时间
→ 保存草稿
→ 发布
```

### 3.1 活动类型

先覆盖四种：

- 秒杀
- 拼团
- 满减
- 折扣

### 3.2 商品绑定

商品绑定必须独立出来，不要把商品、SKU、规则全塞进一个表单对象里。

原因：

- 活动可以绑定多个商品
- 一个商品可以绑定多个 SKU
- 活动商品需要独立查询、编辑和下线

### 3.3 规则配置

规则先以“单活动单规则”为主，避免一开始就做复杂规则编排。

如果后面需要升级，再在 `activity_rule` 上扩展多条规则和优先级字段。

## 4. C 端驱动方式

前端不再靠标题、index 或静态分类猜活动类型，而是按活动主表和规则表驱动。

建议接口返回：

```json
{
  "id": 1,
  "activityName": "618秒杀",
  "activityType": 1,
  "status": 3,
  "startAt": "2026-06-17 10:00:00",
  "endAt": "2026-06-17 12:00:00",
  "products": [],
  "rule": {}
}
```

页面判断：

- 秒杀显示倒计时、库存和抢购按钮
- 拼团显示成团人数、参团按钮和拼团进度
- 满减显示门槛和优惠结果
- 折扣显示折扣率和活动价

## 6. 开发顺序

建议按这个顺序做：

1. 先补活动主表 CRUD
2. 再补活动商品绑定
3. 再补活动规则表
4. 再补 C 端活动详情和列表
5. 最后再做秒杀、拼团、满减、折扣的规则细节

---

## 7. 当前结论

这次先做的不是“大营销中台”，而是活动商品规则的最小闭环。

一句话：

> 先把活动、商品、规则跑通，再谈完整营销引擎。
