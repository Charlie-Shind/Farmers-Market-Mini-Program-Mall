# 最终任务定稿与身份边界

## 1. 最终任务定义

最终交付不是“页面展示”，而是一个可以按流程跑通的农仓平台：

```text
平台配置基础数据
→ C端用户浏览商品/活动
→ C端领取优惠券/参加拼团/秒杀
→ C端下单支付
→ B端商家接单发货
→ C端查看物流/联系客服/确认收货/评价
→ B端回复消息/回复评价/处理售后
→ 平台后台查看全局订单/商户/售后/财务/活动/数据
```

## 2. 三身份定稿

### 2.1 普通用户 USER

用户是 C 端消费者。

权限：

```text
/app/*
```

核心能力：

- 首页、分类、商品、搜索、活动浏览
- 购物车、结算、支付、订单
- 地址、优惠券、积分、收藏
- 售后、评价、客服聊天
- 可申请成为商家

### 2.2 普通商家 MERCHANT

商家是 B 端经营者。

权限：

```text
/merchant/*
```

核心能力：

- 店铺资料和资质
- 商品、SKU、库存、溯源
- 活动、优惠、拼团、秒杀
- 订单接单、发货、售后处理
- 评价管理、消息回复
- 财务流水、提现申请、经营统计

### 2.3 平台管理员 ADMIN

平台管理员是平台 Web 后台运营身份。

权限：

```text
/admin/*
```

核心能力：

- 用户管理
- 商家审核、启停、佣金规则
- 平台商品、类目、商品审核
- Banner、公告、活动、优惠券
- 平台订单、售后仲裁、提现审核
- 平台数据看板
- 平台客服配置

## 3. “平台也是商家”的最终解释

平台也是一个商家，但它不是第四种登录角色。

正确模型：

```text
ADMIN 是平台运营登录身份；
平台官方商家是 Merchant 表里的一条特殊商家数据；
平台商品、平台客服、平台自营订单归属这个官方 Merchant；
普通 MERCHANT 登录后只能操作自己的 Merchant 数据。
```

建议固定数据：

```text
User.openid = platform_merchant
Merchant.storeName = 系统设置 platformOfficialMerchantName，例如“湾源农仓官方店”
SystemSetting.platformSupportMerchantId = 平台官方 Merchant.id
```

## 4. 当前代码里的 P0 身份问题

当前后端 `PlatformDataService.ensureMerchant()` 存在高风险问题：

```text
如果 system_setting.platformSupportMerchantId 存在，ensureMerchant() 会优先返回平台官方商家。
```

后果：

```text
普通商家登录 B 端后，/merchant/products、/merchant/orders、/merchant/refunds 等接口可能全部查到平台官方商家数据。
```

这不符合“三身份 + 平台也是商家”的最终模型。

## 5. 定稿修正原则

必须拆成两个方法：

```ts
ensureCurrentMerchant(authUser)
// 只根据当前登录用户查自己的 Merchant，用于所有 /merchant/* 普通商家接口。

ensurePlatformMerchant()
// 只用于平台官方商品、平台客服、平台配置初始化。
```

禁止：

```text
普通 /merchant/* 接口直接返回平台官方 Merchant
```

允许：

```text
/admin/products 新增平台商品默认归属 ensurePlatformMerchant()
/app/chats/support-target 使用 platformSupportMerchantId
平台官方商品以普通商品形式出现在 C 端
```

## 6. 最终业务边界

| 场景 | 应该使用哪个身份/商家 |
|---|---|
| C端用户下单普通商家商品 | 商品所属 Merchant |
| C端用户下单平台自营商品 | 平台官方 Merchant |
| 普通商家查看订单 | 自己 Merchant 的订单 |
| 平台管理员查看订单 | 全平台订单 |
| 平台客服回复 | 平台官方 Merchant 身份回复 |
| 普通商家客服回复 | 自己 Merchant 身份回复 |
| 平台创建自营商品 | 归属平台官方 Merchant |
| 商家创建商品 | 归属当前商家 Merchant |
