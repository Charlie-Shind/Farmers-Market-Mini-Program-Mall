# 后端定稿约束与必须修改

## 1. 后端目标

后端必须作为三端唯一数据源：

```text
C端只走 /app/*
B端只走 /merchant/*
平台Web后台只走 /admin/*
```

核心交易链路禁止 mock。

## 2. P0：修正商家身份边界

### 当前风险

`backend/src/common/services/platform-data.service.ts` 的 `ensureMerchant()` 当前逻辑会优先返回 `platformSupportMerchantId` 对应的平台官方商家。

这会导致：

```text
普通商家登录 B端 → /merchant/orders → 查到平台官方商家的订单
普通商家发货 → 可能发平台官方订单
商家财务 → 可能查平台官方钱包
```

### 必须改法

新增两个方法：

```ts
async ensureCurrentMerchant(authUser: AuthUser) {
  const user = await this.ensureUser(authUser);
  const merchant = await this.prisma.merchant.findUnique({ where: { userId: user.id } });

  if (!merchant || merchant.deletedAt || merchant.status !== 1) {
    throw new ForbiddenException('商家未入驻或未审核通过');
  }

  await this.prisma.merchantWallet.upsert({
    where: { merchantId: merchant.id },
    create: {
      merchantId: merchant.id,
      availableBalance: new Prisma.Decimal('0.00'),
      frozenBalance: new Prisma.Decimal('0.00'),
      totalIncome: new Prisma.Decimal('0.00'),
      totalWithdrawn: new Prisma.Decimal('0.00'),
    },
    update: {},
  });

  return merchant;
}
```

保留：

```ts
private async ensurePlatformMerchant() { ... }
```

然后把所有普通商家接口中的：

```ts
const merchant = await this.ensureMerchant(authUser);
```

改成：

```ts
const merchant = await this.ensureCurrentMerchant(authUser);
```

涉及方法：

```text
getMerchantProfile
createMerchantProduct
listMerchantProducts
getMerchantProductDetail
updateMerchantProduct
updateProductStatus
updateSkuStock
listMerchantActivities
createActivity/updateActivity/deleteActivity with merchant auth
getMerchantDashboard
listMerchantOrders
getMerchantOrderDetail
acceptOrder
shipOrder
listMerchantRefunds
getMerchantRefundDetail
processRefund
getWallet
getMerchantFinanceRecords
createWithdraw

listMerchantReviews
getMerchantReviewSummary
replyMerchantReview
getMerchantWorkbench
getMerchantStatisticsOverview
getMerchantStatisticsTrend

```

仍然使用 `ensurePlatformMerchant()` 的场景：

```text
/admin/products 创建平台自营商品
/admin/settings 保存 platformOfficialMerchantName 后同步官方店铺
/app/chats/support-target 获取平台客服商户
平台客服会话回复
初始化默认数据
```

## 3. P0：订单状态机定稿

后端状态机必须固定：

```ts
ORDER_STATUS = {
  PENDING: 1,     // 已下单，待商家接单。未支付订单也可以是 PENDING，但 payStatus=0。
  ACCEPTED: 2,    // 商家已接单
  COMPLETED: 3,   // 用户已确认收货
  CANCELLED: 4,
}

PAY_STATUS = {
  UNPAID: 0,
  PAID: 1,
}

DELIVERY_STATUS = {
  NONE: 0,        // 未发货
  TO_SHIP: 1,     // 商家已接单，待发货
  SHIPPED: 2,     // 已发货，待收货
}

REFUND_STATUS = {
  NONE: 0,
  PENDING_MERCHANT: 1,
  PROCESSING: 2,
  APPROVED: 3,
  REJECTED: 4,
}
```

状态流转：

```text
创建订单：orderStatus=1, payStatus=0, deliveryStatus=0
支付成功：orderStatus=1, payStatus=1
商家接单：orderStatus=2, deliveryStatus=1
商家发货：orderStatus=2, deliveryStatus=2
用户确认收货：orderStatus=3, deliveryStatus=2
申请售后：refundStatus=1
商家处理：refundStatus=3 或 4
取消订单：orderStatus=4
```

所有前端按钮都以这个状态机判断。

## 4. P0：父订单号与子订单号

普通购物车跨商家下单会有父订单和子订单：

```text
parentOrderNo：支付用
childOrderNo：订单详情、商家接单、发货、物流、确认收货、评价、售后用
```

后端 `POST /app/orders` 必须返回：

```json
{
  "orderNo": "父订单号",
  "payAmount": "98.00",
  "childOrderNos": ["子订单号1", "子订单号2"]
}
```

C 端支付用 `orderNo`，跳详情用 `childOrderNos[0]`。

秒杀、拼团如果只生成单个商家订单，可以没有父子拆分，但建议也返回：

```json
{
  "orderNo": "订单号",
  "payAmount": "68.00",
  "childOrderNos": ["订单号"],
  "orderMode": "FLASH_SALE"
}
```

## 6. P0：评价接口合同

### C端提交评价

```http
POST /app/orders/:orderNo/reviews
```

Body 必须是数组包装：

```json
{
  "reviews": [
    {
      "orderItemId": 1,
      "rating": 5,
      "content": "商品新鲜，发货快",
      "images": []
    }
  ]
}
```

后端约束：

```text
1. 只能评价自己的订单。
2. 订单必须 COMPLETED。
3. 一个 orderItemId 只能评价一次。
4. content 不能为空。
```

### B端评价列表

```http
GET /merchant/reviews?page=1&pageSize=20&hasReply=false&rating=5
```

返回：

```ts
AppPagedResult<{
  id: number;
  orderNo: string;
  buyer: { userId: number; nickname: string; avatarUrl: string };
  product: { productId: number; title: string; coverUrl: string; skuName: string };
  rating: number;
  content: string;
  images: string[];
  replyContent: string | null;
  repliedAt: string | null;
  createdAt: string;
}>
```

### B端回复评价

```http
POST /merchant/reviews/:reviewId/reply
```

Body：

```json
{
  "content": "感谢支持，我们会继续保证品质。"
}
```

后端建议兼容旧字段：

```ts
const content = String(body.content ?? body.replyContent ?? '').trim();
```

## 8. P1：秒杀定稿

秒杀必须走：

```text
GET /app/quick/flash-sale/items
→ POST /app/quick/flash-sale/claim
→ POST /app/orders/preview with orderMode=FLASH_SALE
→ POST /app/orders with orderMode=FLASH_SALE
→ 支付
```

秒杀订单 Body：

```json
{
  "orderMode": "FLASH_SALE",
  "flashSaleItemId": 1,
  "skuId": 1,
  "quantity": 1,
  "addressId": 1,
  "couponId": null,
  "usePoints": 0,
  "remark": "秒杀订单"
}
```

后端必须用 `FlashSaleItem.flashPrice`，不能用 SKU 原价。

## 10. P1：促销价定稿

促销价在 SKU 层：

```prisma
promotionPrice Decimal?
promotionStartAt DateTime?
promotionEndAt DateTime?
```

所有商品列表、详情、订单预览、订单创建必须使用服务端计算：

```ts
displayPrice = now in [promotionStartAt, promotionEndAt] ? promotionPrice : price
```

禁止只在前端展示促销价但订单仍按原价。

## 11. 后端编译约束

当前包如果出现：

```text
sh: 1: nest: Permission denied
```

修复：

```bash
cd backend
chmod +x node_modules/.bin/*
npm run build
```

正式开发机器建议重新安装依赖：

```bash
rm -rf node_modules package-lock.json
npm install
npx prisma generate
npm run build
```
