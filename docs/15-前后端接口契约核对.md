| `POST /api/auth/wechat/phone-bind` | 已有 | `phoneCode` 只作为换号凭证，后端服务端换取真实手机号后写入 `mobile`，并返回更新后的用户摘要 |

约束说明：

- `accountNo` 是业务唯一号，不替代数据库主键 `id`
- `accountNo` 按“日期 + 当日序号”生成，前缀区分用户和管理员
- 前端和客户端不得自行拼接 `accountNo`
- 访客 token 不得共享固定匿名用户记录，每次签发都应视为独立会话
- 已禁用账号不能通过微信登录、手机号绑定或游客升级链路重新变成有效会话
- 手机号授权链路必须以微信官方换号接口结果为准，不能把 `phoneCode` 当作手机号直接落库
- 当后端缺少微信小程序配置时，接口只返回通用失败信息，不暴露内部配置细节

## 6. 路由与契约不一致点

### 6.1 登录接口命名不一致

当前仓库里有两套命名：

- 文档约定：`/api/auth/guest`、`/api/auth/wechat/login`、`/api/auth/wechat/phone-bind`
- 后端实际：`/api/identity/auth/anonymous`、`/api/identity/auth/status`、`/api/identity/auth/me`

这会导致前端和文档同时存在时出现理解偏差。

当前已经明确的处理方式是：

- 对外契约仍以 `auth` 路由为主，兼容旧路由逐步收敛
- 账号唯一号统一使用 `accountNo`
- 微信手机号绑定统一按服务端换号流程实现

### 6.2 权限声明不完整

后端已经有全局 `JwtAuthGuard` 和 `RolesGuard`，但目前业务 controller 没有显式补 `@Roles(...)`。

结果是：

- 路由表面上“受保护”
- 实际上很多非 public 路由只是在校验 token 是否存在
- 角色级别约束没有真正生效

### 6.3 部分状态是占位枚举

例如：

- `PENDING_AUDIT`
- `PENDING_ARBITRATION`
- `TO_SHIP`
- `DRAFT`

这些状态当前在多个模块里可以返回，但有些并没有对应完整状态机。

### 6.4 异常返回口径

- 非 `HttpException` 的服务端异常统一返回通用 `Internal server error`
- 不对前端暴露数据库、堆栈或上游服务细节
- 业务错误仍然通过 `message` 传达，但不能把实现细节写进去
