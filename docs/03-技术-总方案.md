# 03-技术-总方案

> 项目技术总方案，统一说明选型、架构、模块拆分、数据关系、部署与第一阶段范围。

## 1. 项目目标

浔源农仓是一个农产品电商平台，目标覆盖：

- C 端用户浏览、下单、支付、售后
- B 端商家入驻、商品管理、发货、财务
- 平台后台审核、配置、仲裁、看板
- 后续扩展营销、积分、分销、团长、自提点

第一阶段只做核心交易闭环：

- 商品展示与分类
- 购物车
- 下单与支付
- 订单管理与发货
- 售后处理
- 商家入驻与商品审核
- 平台后台管理

---

| ORM | Prisma |

## 3. 总体架构

```text
前端
├── C端用户小程序
├── B端商家小程序
└── 平台管理端

↓ HTTPS + JSON API

后端服务 (NestJS)
├── auth            登录、JWT、角色权限
├── user            用户、地址、收藏、积分
├── merchant        商家入驻、资质、钱包、提现
├── product         分类、商品、SKU、图片、溯源
├── cart            购物车
├── order           订单、拆单、状态流转
├── payment         微信支付、回调、流水
├── refund          售后申请、处理、仲裁
├── marketing       优惠券、活动、积分
├── file            文件上传、对象存储
└── distribution    团长、分佣、自提点

↓ 数据层

PostgreSQL 业务数据
Redis 登录 Token、库存锁、订单超时、缓存
对象存储 图片、视频、资质文件
```

## 4. 模块划分

### 4.1 Auth

- 微信登录
- 匿名访客 Token（guest）
- JWT 签发与校验
- 角色权限控制

说明：

- 未登录访客可以先获得临时身份 `guest`
- `guest` 只用于浏览、临时购物车、临时收藏等弱身份能力
- 登录后由 `guest` 升级为正式用户身份 `user`
- 临时数据在升级时需要合并到正式账号

### 4.2 User

- 用户信息
- 地址管理
- 收藏商品
- 积分、优惠券
- 订单查询
- 售后申请

### 4.3 Merchant

- 商家入驻
- 资质上传
- 店铺管理
- 商品管理
- 库存管理
- 订单接单与发货
- 售后处理
- 营收与提现

补充约定：

- 当前不对外开放独立 B 端商家运营入口
- 平台默认存在一个官方商户账号，用于承接平台商品、官方客服和后台默认归属
- 官方账号名称由系统设置控制，默认值为 `浔源农仓`

### 4.4 Product

- 商品分类
- 商品发布、SKU 管理
- 图片、视频、溯源资料
- 审核与上下架

### 4.5 Cart

- 加入购物车
- 修改数量
- 勾选与删除
- 购物车结算

### 4.6 Order

- 创建订单
- 金额计算
- 状态管理
- 多商家拆单
- 取消与完成

### 4.7 Payment

- 微信支付下单
- 微信支付回调
- 签名验证
- 支付流水记录
- 退款处理

### 4.8 Refund

- 用户售后申请
- 商家处理
- 平台仲裁
- 退款、退货、佣金回滚

### 4.9 Admin

- 管理员登录
- 用户管理
- 商户审核
- 商品审核
- 订单管理
- 售后管理
- Banner、活动、类目、运费模板配置
- 数据看板

### 4.10 Marketing

- 优惠券
- 满减活动
- 活动商品关联
- 积分兑换

补充约定：

- 营销能力优先收敛到一个 `marketing` / `promotion` 主模块里，先把创建、发布、下线、校验这些主流程集中处理
- 不要为了“架构感”过早拆成很多业务文件，只有价格计算、库存、状态机这类高耦合/高复杂逻辑再单独抽出少量服务
- 秒杀、拼团、促销可以统一入口，但不同活动的执行逻辑仍要保持各自清晰，避免后面维护时到处找文件
- 当前优先级先做“活动 + 商品 + 规则”三件事，也就是先把活动主表、活动商品绑定、活动规则配置这条链路打通，再考虑更完整的营销引擎
- 活动页和商家后台都应以 `activityId` / `promotionId` 为主线，不再依赖标题、索引或前端静态分类猜测业务

### 4.11 Distribution

- 团长申请与审核
- 团长邀请码/小程序码
- 用户绑定团长关系
- 分佣规则与佣金记录
- 提现管理

## 6. 关系与约束

- `user` 1:N `user_address`
- `user` N:M `role` via `user_role`
- `merchant` 1:N `product`
- `product` 1:N `product_sku`
- `product` 1:N `product_image`
- `product` 1:N `product_video`
- `product` 1:N `product_trace`
- `user` 1:N `cart`
- `user` 1:N `orders`
- `orders` 1:N `order_item`
- `orders` 1:N `payment_record`
- `orders` 1:N `delivery_record`
- `orders` 1:N `refund_apply`
- `user` 1:N `user_coupon`
- `coupon` 1:N `user_coupon`
- `community_leader` 1:N `commission_record`

必须保留快照字段，避免商品、地址、价格后续变更污染历史单据。

## 7. 状态枚举建议

- 用户状态：正常 / 禁用 / 注销
- 商家审核状态：待提交 / 待审核 / 审核通过 / 拒绝 / 已停用
- 商品状态：草稿 / 待审核 / 上架 / 下架 / 审核拒绝
- 订单状态：待支付 / 已支付 / 待接单 / 待发货 / 已发货 / 已完成 / 已取消 / 已关闭
- 售后状态：申请中 / 商家处理中 / 平台处理中 / 已同意 / 已拒绝 / 已退款 / 已关闭
- 支付状态：待支付 / 支付中 / 成功 / 失败 / 已退款
- 提现状态：待审核 / 已通过 / 已拒绝 / 已打款 / 作废

建议状态值在服务层统一定义，数据库只存整数。

## 11. 核心设计原则

1. 先做核心闭环，再做扩展模块。
2. 金额字段统一用 `numeric(18,2)`。
3. SKU 规格用 `jsonb` 或等价 JSON 字段。
4. 图片、视频、文件只存 URL，不存二进制。
5. 订单、支付、售后必须保留快照。
6. 支付、退款、提现必须具备幂等能力。
7. 权限控制采用 NestJS Guards + JWT + Role。
8. 匿名访客使用独立的 `guest` 身份，不等同于正式用户登录态。
9. 业务文件优先集中，主流程先放在同一个模块里，只有真正复杂的能力再拆分。
