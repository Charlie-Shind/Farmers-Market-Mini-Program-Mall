# 图标清单（详细版）

> 用途：把项目里**所有需要用到的图标**按"现状 vs 需求"整理成一张可执行的工程交付表。
> 配合 [`16-图标清单与版权授权.md`](./16-图标清单与版权授权.md) 一并使用——那份是版权/采购指南，本份是工程交付清单。
>
> 维护原则：
> - 新增页面 / 模块时，**先**对照本表确认是否复用已有图标。
> - 必须新增时，**再**回头补一行到本表"缺漏清单"。
> - 图标入库后，把"已有/缺漏"列翻转，**新增**栏目追加到"已有图标索引"。

### 3.3 命名/语义错位（必须决策后再补）

| 问题 | 现状 | 影响 | 建议处理 |
|---|---|---|---|
| C 端 `package` 已被代码引用作 B 端底部导航的"商品" | 实际 SVG 是包裹盒 | 业务语义混淆 | 新增 `b-product`，把 B 端导航的 `package` 改键名 |
| C 端 `truck`/`wallet`/`shield` 用作 B 端"物流/财务/保障" | 语义勉强对应但放在 B 端导航含义不精准 | 用户识别成本 | 评估是否重命名为 `b-delivery` / `b-finance` / `b-shield` |
| `refund` 与"售后" | icons.ts 已声明 `refund` | 与新加的 `order-refund` 重复 | 合并为 `refund` |
| `message` vs `bell` | `message` 偏"信封" | 通知红点更习惯用 bell | 评估替换或并存（建议并存：消息=会话、铃铛=通知） |
| B 端底部导航 vs 现有 C 端键名 | C 端有 `home/category/cart/profile`，B 端需要不同入口 | 复用了 `home` | 给 B 端统一加 `b-` 前缀 |

## 5. 命名规范

### 5.1 通用约定

- **kebab-case** 命名（与现有 `icons.ts` 一致）：`order-pending-pay`、`b-home`、`message-success`
- 单图标的英文动词用原形：`-on`/`-off`、`-asc`/`-desc`
- 同语义的多个变体用后缀：`-filled` / `-outline`、`-light` / `-dark`
- 平台前缀：`b-`（B 端）、`cat-`（分类图）、`weather-`（天气）……只有 C 端通用图标不需前缀

### 5.2 平台归属

| 前缀 | 平台 | 出现位置 |
|---|---|---|
| 无前缀 | C 端通用 / 全平台 | `wxapp/miniprogram` |
| `b-` | B 端商家专用 | `wxapp/miniprogram`（同包，B 端页面） |
| `cat-` | 商品分类业务图 | `wxapp/miniprogram` |
| `admin-` 或后缀 `-admin` | 后台专用 | `admin/` |

> C/B 端 SVG 全部放在 `wxapp/miniprogram/assets/icons/`，Admin 放 `admin/src/assets/icons/`。
> `wxapp/miniprogram/config/icons.ts` 与 `admin/src/assets/icons.ts` 各自维护一份。

### 5.3 入库流程

1. **申请** → 从 `iconfont.cn` / `Lucide` / `Iconify` 拉到 SVG
2. **重命名** → 按 5.1 规范命名，存到对应平台目录
3. **注册** → 在 `config/icons.ts`（C/B）或 `admin/src/assets/icons.ts`（Admin）追加 key
4. **核对** → 同步更新本表第 1 节"已有图标索引"
5. **引用** → 在业务代码 `import { iconPaths }` 拿路径

## 7. 验收方式

新图标落地后，必须同时满足：

1. **文件存在**：`assets/icons/<key>.svg` 实际存在
2. **类型声明**：`IconName`（`config/icons.ts` 中 `keyof typeof iconPaths`）覆盖新 key
3. **设计稿替换**：原占位 unicode 字符在对应页面已删除
4. **跨平台一致**：C/B 端同语义图标形态一致
5. **本表更新**：第 1 节"已有" + 第 3 节"缺漏" 状态翻转
