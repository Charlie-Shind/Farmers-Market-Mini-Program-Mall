## 1. 当前结论

现在已经可以开始搭后端，而且仓库里已经有了可运行的 NestJS 骨架和 Prisma 落地结构。  
但整体仍属于“**可启动、未完全闭环**”的状态。

原因是：

- 核心业务链路已经明确
- 数据表设计已经覆盖主流程
- 接口草案已经有基础
- 后端工程骨架已经创建
- 但真实业务逻辑、部分扩展表和最终联调还没全部闭合

仓库里现在已经能看到后端项目骨架，例如：

- `backend/package.json`
- `backend/src/`
- `backend/prisma/`
- `backend/nest-cli.json`

也就是说，现在已经不是纯文档阶段，而是**文档 + scaffold 并行推进阶段**。

## 3. 目前还不完善的点

### 3.1 身份与权限边界还不够细

当前文档已经提到了用户、商家、管理员三类身份，但还没有把以下内容彻底闭合：

- 未登录访客是否需要 `guest` 临时身份
- `guest` 的能力边界是什么
- C 端登录后如何识别是否为 B 端用户
- B 端用户和普通用户是否共享同一个 `user`
- 管理后台是否独立账号体系
- 各接口的鉴权规则如何统一
- 角色权限如何映射到菜单、路由和 API

建议的补充规则：

- `guest` 只用于浏览、临时购物车、临时收藏等弱身份场景
- 登录态与访客态共用同一套 JWT 验证框架，但 `type` 或 `role` 必须区分
- 登录后可以将 `guest` 的临时数据迁移到 `user`
- 权限判断不只依赖 token 字段，涉及封禁和变更的内容应回查数据库或 Redis
- 访客 token 不应共享单一匿名用户记录，否则会把不同会话的购物车、收藏和浏览数据混在一起

### 3.2 后端接口还缺统一约定

接口文档已经有雏形，但还缺一些工程级规则：

- 列表分页参数统一规范
- 排序参数统一规范
- 查询过滤参数统一规范
- 返回体统一格式
- 错误码体系
- 文件上传返回结构
- 审核类接口的状态流转约定

### 3.3 后台管理接口不够完整

平台后台目前有：

- 登录
- 商户审核
- 商品审核
- Banner 配置
- 活动配置
- 订单查询
- 售后仲裁
- 数据看板
- 文件上传

但还缺：

- 当前登录管理员信息接口
- 菜单权限接口
- 角色管理接口
- 用户管理接口
- 商户列表与筛选接口
- 配置项 CRUD
- 操作日志接口

补充说明：

- `GET /api/admin/users` 已经具备分页、关键词和状态筛选能力，并应返回真实的积分、订单数和最近登录时间
- `PATCH /api/admin/users/:userId/status` 已补上，后端负责用户启停，不允许前端自行篡改用户状态
- 用户列表不要把聚合字段交给前端猜测，避免后台和前端统计口径漂移
- 已禁用账号在 C 端登录和手机号绑定链路中都应被拦截，不能继续复用为有效会话

### 3.4 ORM 还只覆盖核心交易链路

第一版 Prisma 已经能跑通核心交易和后台账号体系，但还不足以支撑完整平台功能。  
需要补强的通常是：

- 配置模型
- 活动/优惠/积分的扩展模型
- 更完整的退款/结算/财务模型

### 3.5 工程基础设施还没落地

后端开发前还需要确认：

- 环境变量规范
- 配置加载方式
- 数据库迁移策略
- 日志与审计方案
- 接口文档生成方式
- 开发、测试、生产环境区分
- 文件上传存储方式
- Redis key 命名规范

### 3.6 测试与发布流程未定义

目前还没有明确：

- 单元测试怎么写
- 集成测试怎么做
- 核心接口怎么验收
- 支付回调怎么模拟
- 上线前检查项是什么

## 4. 先补哪些最划算

建议优先补下面这几项：

1. 登录与鉴权规则
2. 访客 `guest` 身份与登录升级规则
3. 后台账号与权限模型
4. API 通用返回规范
5. 分页与筛选规范
6. Prisma 初始表结构
7. 后端项目骨架
8. 核心模块的最小接口集合

---

## 5. 不建议第一轮就做的内容

第一轮不要急着做：

- 优惠券全规则
- 积分体系
- 拼团秒杀完整引擎
- 分销和团长分佣
- 复杂财务结算
- 复杂售后仲裁流

这些都依赖订单闭环稳定之后再做。

---

## 6. 结论

当前后端的状态是：

- 方案基本齐了
- 数据模型基本齐了
- 接口草案基本齐了
- 工程骨架已经有了
- 但真实业务流程还在补齐

所以现在最合适的做法不是继续扩散需求，而是先搭后端骨架，优先把核心闭环跑起来。

当前已明确的稳定契约包括：

- 访客身份按独立会话签发，不共享单一匿名记录
- 后端 500 错误统一返回通用消息，不暴露内部实现细节
- 管理端用户列表的积分和订单数由后端汇总提供
