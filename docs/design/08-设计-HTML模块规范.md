# 08-设计-HTML模块规范

## 1. 文档目的

本文档用于统一“设计参考稿”阶段的 HTML 模块表达方式，避免页面结构、命名和注释风格不一致。

适用范围：

- 设计参考页
- 原型页
- 静态 HTML 预览页
- 模块拆分说明页

本文档不约束业务实现代码，只约束设计展示层的写法和组织方式。

---

## 2. 设计原则

1. **先结构，后样式**  
   先定义页面模块顺序、信息层级和交互关系，再补充视觉表现。

2. **模块化表达**  
   每个功能区块必须有明确的模块边界，便于后续拆分和复用。

3. **参考优先**  
   当前目录内容均视为设计参考，不直接代表最终开发实现。

4. **语义清晰**  
   HTML 标签优先使用语义化结构，避免大量无意义嵌套。

5. **可复用、可扩展**  
   同类内容应抽象成统一模块写法，例如分类入口、筛选标签、商品卡片、导航栏。

---

## 3. HTML 文件规范

### 3.1 基本要求

- 文件编码统一为 `UTF-8`
- 页面语言统一使用 `lang="zh-CN"`
- 必须包含 `meta viewport`
- 页面标题应与模块主题一致
- 设计稿应保留为单页可预览结构

### 3.2 文件结构建议

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
  <title>页面标题</title>
  <link rel="stylesheet" href="./微信小程序/C端/01-首页.css">
</head>
<body>
  <div class="page">
    <!-- 页面模块按顺序排列 -->
  </div>
</body>
</html>
```

### 3.3 推荐结构顺序

1. 顶部状态栏
2. 顶部操作区
3. 首屏核心区域
4. 功能模块区
5. 推荐列表 / 内容流
6. 底部导航

---

## 4. 模块设计规范

### 4.1 模块定义

每一个功能区块应被视为独立模块，例如：

- 首页 Banner 模块
- 商品分类模块
- 筛选标签模块
- 商品卡片模块
- 底部导航模块

模块内部结构应保持稳定，便于后续复用到其他页面。

### 4.2 模块命名

建议统一使用以下命名方式：

- 页面容器：`page`
- 模块区块：`section`、`hero`、`category-module`
- 模块头部：`section-head`
- 卡片容器：`card`、`product-card`
- 导航容器：`bottom-nav`
- 标签组件：`tag`

命名要求：

- 使用英文小写
- 单词之间使用连字符或驼峰保持一致
- 不使用无意义缩写
- 同类组件保持同一命名风格

### 4.3 模块注释

每个模块必须带简短注释，说明模块职责。

推荐格式：

```html
<!-- 5. 商品分类：一级类目入口 + 复用筛选标签 -->
<section class="section">
  ...
</section>
```

注释要求：

- 简短
- 可读
- 对应文档章节编号
- 说明模块用途，不描述样式细节

## 6. 模块编写标准

### 6.1 分类模块

分类模块建议拆成两部分：

- 一级类目入口
- 筛选标签

说明：

- 一级类目用于进入对应分类列表
- 筛选标签用于缩小商品范围
- 筛选标签可复用到搜索页、推荐页、活动页

### 6.2 商品卡片模块

商品卡片建议至少包含：

- 商品图片或视觉占位
- 商品标题
- 简要说明
- 价格信息
- 辅助信息

如果用于设计参考，可使用占位图形或渐变块代替真实商品图。

### 6.3 导航模块

底部导航应保持固定、清晰、低干扰。

建议包含：

- 首页
- 分类
- 购物车
- 我的

## 7. 样式规范

### 7.1 CSS 组织原则

- 使用 `:root` 统一定义颜色、圆角、阴影等变量
- 组件样式尽量独立，减少全局污染
- 复杂页面按模块分段书写
- 颜色、字体、间距保持统一体系

### 7.2 视觉建议

- 主色建议稳定在农产品、自然、清爽的视觉方向
- 卡片、按钮、标签应有清晰层级
- 页面不要过度堆叠装饰，重点突出信息层次
- 移动端优先，适配窄屏展示

### 7.3 响应式要求

- 页面应兼容移动端展示
- 小屏幕下模块允许换行、压缩和堆叠
- 关键区域不能依赖固定宽度

## 8. 数据与文案规范

### 8.1 数据展示

设计页中的数据只作为展示样例使用，例如：

- 商品名称
- 价格
- 销量
- 标签
- 提示文案

### 8.2 文案要求

- 文案简洁明确
- 重点突出业务场景
- 避免过度口语化
- 避免用实现描述代替产品描述

---

## 9. 复用与扩展规范

### 9.1 可复用组件

以下内容建议抽象为通用组件：

- 标签组件
- 商品卡片
- 分类卡片
- 信息提示条
- 模块标题栏
- 底部导航

### 9.2 扩展原则

新增模块时应遵循：

1. 先补充文档说明
2. 再补 HTML 结构
3. 最后补 CSS 样式
4. 保持与现有风格统一

## 11. 建议写法模板

模块描述建议按以下格式书写：

```md
### 1.X 模块名称（模块目标）

- 功能点 1：……
- 功能点 2：……
- 功能点 3：……

我们理解为：

- 说明这个模块在页面中的职责
- 说明它和其他模块的关系
- 说明它是否可复用
```

## 12. 最终要求

后续所有 HTML 设计参考稿，应统一遵守以下要求：

- 有明确的模块编号
- 有清晰的模块标题
- 有对应的注释说明
- 有统一的命名规范
- 有可复用的结构设计
- 有适配移动端的布局思路

如果后续补充新的页面类型，建议先在本规范下增加对应模块说明，再进入 HTML 设计。
