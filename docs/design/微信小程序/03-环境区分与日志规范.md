# 03-环境区分与日志规范

> 适用范围：微信小程序开发、联调、测试、正式发布三个阶段。  
> 说明：本规范只描述微信小程序侧最小可落地的环境识别、请求配置和日志分级，不替代后端日志体系。

---

## 1. 目标

这个规范只解决三件事：

- 识别当前小程序处于开发版、体验版还是正式版
- 将请求地址、调试开关、日志级别统一收口
- 避免调试代码和测试配置误入正式环境

---

## 2. 环境分层

### 2.1 微信运行态环境

微信小程序运行时只区分三种发布形态：

- `develop`：开发版
- `trial`：体验版
- `release`：正式版

建议通过 `wx.getAccountInfoSync().miniProgram.envVersion` 读取当前运行态。

### 2.2 项目构建环境

项目构建环境建议由前端工程自己控制，常见取值：

- `dev`
- `test`
- `prod`

这个环境用于：

- 选择接口域名
- 选择日志级别
- 选择是否打开调试开关

### 2.3 两类环境的区别

- 微信运行态环境：描述“小程序现在以什么形式运行”
- 构建环境：描述“这次构建要连哪个服务、开哪些调试能力”

二者不要混用。

---

## 3. 配置策略

### 3.1 配置原则

- 运行时不要把环境写死在页面里
- 请求域名统一放在配置文件里
- 页面组件只关心业务，不关心环境判断

### 3.2 推荐结构

```txt
miniprogram/
  config/
    env.ts
    index.ts
  services/
    request.ts
  utils/
    logger.ts
```

### 3.3 配置职责

- `config/env.ts`：读取构建环境和运行态环境
- `config/index.ts`：导出 API Base、日志开关等统一配置
- `services/request.ts`：封装请求实例并读取配置
- `utils/logger.ts`：统一输出日志

---

## 4. 环境识别

### 4.1 运行态识别

```ts
const accountInfo = wx.getAccountInfoSync();
const runtimeEnv = accountInfo.miniProgram.envVersion; // develop | trial | release
```

### 4.2 构建环境识别

建议构建时注入：

```ts
export const APP_ENV = 'dev' as 'dev' | 'test' | 'prod';
```

或者通过构建脚本生成环境文件，不建议在页面内直接写常量。

### 4.3 使用原则

- `runtimeEnv` 只用于判断当前发布形态
- `APP_ENV` 只用于选择接口、日志和调试策略

---

## 5. 请求配置

请求层统一管理：

- `baseURL`
- `token`
- 超时
- 错误提示
- 登录失效处理

### 5.1 基本要求

- 页面不直接写 `wx.request`
- 所有请求走统一封装
- 业务模块只调用 `request` 或接口方法

### 5.2 示例

```ts
export const API_BASE =
  APP_ENV === 'dev'
    ? 'http://localhost:6002/api'
    : APP_ENV === 'test'
      ? 'https://test-api.example.com/api'
      : 'https://api.example.com/api';
```

---

## 6. 日志分级

### 6.1 日志级别

- `debug`：仅开发环境输出
- `info`：开发和测试环境输出
- `warn`：所有环境保留
- `error`：所有环境保留

### 6.2 推荐封装

```ts
export const logger = {
  debug: (...args: unknown[]) => {
    if (APP_ENV === 'dev') console.debug('[DEBUG]', ...args);
  },
  info: (...args: unknown[]) => {
    if (APP_ENV !== 'prod') console.info('[INFO]', ...args);
  },
  warn: (...args: unknown[]) => console.warn('[WARN]', ...args),
  error: (...args: unknown[]) => console.error('[ERROR]', ...args),
};
```

### 6.3 输出原则

- 调试日志不要污染正式环境
- 错误日志要保留关键信息
- 不要在日志里输出敏感字段明文

---

## 7. 调试建议

- 开发版：用于页面联调和接口联调
- 体验版：用于 QA 验收
- 正式版：只保留必要错误日志

如果需要集中排查问题，可以再补一层：

- 后端日志接口
- 远程错误上报
- 用户操作追踪

但这三项不属于当前最小闭环，后续再补。

---

## 8. 结论

最小可落地方案就是：

1. 用 `wx.getAccountInfoSync().miniProgram.envVersion` 识别运行态
2. 用构建环境控制 `API_BASE` 和日志级别
3. 统一封装 `request` 和 `logger`
4. 不在页面层手写环境分支
